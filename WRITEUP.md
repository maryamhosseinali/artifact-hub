# Artifact Hub — Writeup

## What I built

A small full-stack system for publishing, browsing, reviewing, and sharing
AI-generated artifacts (HTML, images, PDFs):

- **Publish flow** — upload a file + optional title/description/tags. Omitted
  metadata fields are auto-generated by Claude from the artifact's actual
  content (HTML source, image, or PDF — all sent as native multimodal content
  blocks, not just a filename guess).
- **Gallery** — filterable by type/tag, plus a free-text search box backed by
  an LLM re-rank over title/description/tags.
- **Artifact detail page** — renders the artifact, shows tags, lets people
  comment, generate a feedback summary, and create a share link.
- **Comments** — flat list, author name + body + timestamp, no auth.
- **Share links** — `POST /api/artifacts/:id/share` mints a cryptographically
  random token (`crypto.randomBytes(32)`, not a sequential/guessable id) with
  a caller-chosen expiry (60s–30d); `/share/:token` resolves it server-side and
  enforces `expiresAt` on every view, returning a distinct "expired" state vs.
  "not found."
- **MCP server** — a Streamable HTTP server (`@modelcontextprotocol/sdk`)
  exposing 5 tools that call directly into the same `packages/core` logic the
  web app uses: `publish_artifact`, `list_artifacts`, `search_artifacts`,
  `share_artifact`, `get_feedback_summary`.
- **Two LLM features**, both usable from the web UI and from MCP:
  1. Auto-title/description/tags at publish time when fields are omitted.
  2. On-demand feedback summarization across an artifact's comments (themes,
     action items, disagreements).

## What I explicitly did not build (and why)

- **Multi-user auth / RBAC** — single shared workspace, one static bearer
  token gates the MCP server. Out of scope for the timebox; the interesting
  parts of this exercise (MCP integration, LLM enrichment, share-link
  mechanics) don't need it, and building real auth well is its own project.
- **OAuth for MCP** — a static bearer token is enough to demonstrate the
  MCP integration; wiring a full OAuth flow would have eaten the day without
  changing what's being evaluated.
- **Real-time collaboration/notifications, artifact version history, rich
  comment editing/threading/reactions** — all explicitly out of scope per the
  brief; flat comments and a simple publish flow are enough to exercise the
  full system end-to-end.

## Architecture

Monorepo (npm workspaces) so the web app and MCP server share one source of
truth for business logic instead of duplicating it behind an internal HTTP
call:

```
apps/web/         Next.js 16 (App Router) — gallery, viewer, comments, share UI, REST API routes
apps/mcp-server/   Standalone Node service — MCP Streamable HTTP server
packages/core/     Prisma client + artifact/comment/share-link logic + Anthropic calls — shared by both apps
```

`apps/web`'s API routes and `apps/mcp-server`'s tool handlers both import
directly from `packages/core` (e.g. `createArtifact`, `listArtifacts`,
`createShareLink`). Neither app talks to the other over HTTP — the "shared
backend" is a shared library, not a second service to keep in sync. This is
also why the MCP layer is more than CRUD wrappers: the interesting logic
(LLM enrichment, token generation, expiry enforcement) lives in one place,
and the MCP tool handler is a thin adapter over it, same as a route handler
is.

### Data model (Prisma / Postgres)

```
Artifact(id, title, description, type, fileUrl, tags[], sourceTool, createdAt)
Comment(id, artifactId, authorName, body, createdAt)
ShareLink(id, artifactId, token, expiresAt, createdAt)
```

### Stack

- Next.js 16 + Tailwind, deployed on Vercel
- Postgres via Prisma — Neon
- File storage — Vercel Blob
- MCP server — separate long-running Node process (Express + the MCP SDK's
  `StreamableHTTPServerTransport`), deployed on Railway via the root
  `Dockerfile`

## A security decision worth calling out: sandboxing HTML artifacts

This system stores and serves arbitrary (often LLM-generated) HTML. Rendering
that HTML directly on the app's own origin would let any artifact read the
viewer's cookies and call the app's API as them — a same-origin XSS hole.
Two things avoid it here, deliberately:

1. HTML artifacts are stored in Vercel Blob, a different origin from the web
   app. The browser fetches the artifact's HTML directly from blob storage,
   not from `this-app.vercel.app`.
2. The artifact viewer (`components/ArtifactViewer.tsx`) renders HTML in an
   `<iframe sandbox="allow-scripts">` — deliberately **without**
   `allow-same-origin`. The artifact can run script, but that script cannot
   read the app's cookies, call the app's API as an authenticated user, or
   navigate the top-level page.

Images render as `<img>`; PDFs render in a plain iframe (browsers' built-in
PDF viewers don't execute arbitrary script from the PDF's origin the way an
HTML document would).

## Why MCP is hosted as a separate long-running service, not on Vercel

The MCP TypeScript SDK's `StreamableHTTPServerTransport` keeps per-session
state (the session-id → transport map) in memory, tied to one process. Vercel
functions are stateless and can be routed to different instances between
requests, which would silently break a multi-request MCP session (initialize
on instance A, next call lands on instance B with no memory of the session).
Rather than fight that with the SDK's stateless mode (which would limit what
the transport can do across a session), I ran the MCP server as its own
always-on Node process on Railway — a normal Express app holding an in-memory
`transports` map exactly as the SDK's own reference example does. Two
deployments and two URLs instead of one, but no serverless/statefulness
footgun to explain away later.

## MCP auth

Single static bearer token, checked via the MCP SDK's `requireBearerAuth`
middleware with a trivial token verifier (no OAuth). A reviewer connects via:

- **Claude Desktop → Settings → Connectors → Add custom connector** — paste
  the server URL, paste the token under Advanced settings.
- **Manual `claude_desktop_config.json`:**
  ```json
  {
    "mcpServers": {
      "artifact-hub": {
        "url": "https://<mcp-server>.up.railway.app/mcp",
        "headers": { "Authorization": "Bearer <token>" }
      }
    }
  }
  ```
- Or via `mcp-remote` if the client doesn't support custom headers directly:
  ```
  npx -y mcp-remote https://<mcp-server>.up.railway.app/mcp --header "Authorization: Bearer <token>"
  ```

## Where and why the LLM is used

Both features call the Anthropic API (`claude-opus-4-8`, adaptive thinking)
server-side only — the API key never reaches the client, and both the MCP
server and the web app's route handlers call the same `packages/core/src/llm.ts`
functions:

1. **Auto-metadata at publish time** (`generateArtifactMetadata`) — when
   title/description/tags are omitted, the artifact's actual content is sent
   to Claude as a native content block (raw HTML text, an `image` block, or a
   `document` block for PDFs — not just a filename), constrained to a JSON
   schema via `output_config.format`, so the response is guaranteed-valid
   JSON with no parsing games.
2. **Feedback summarization** (`summarizeFeedback`) — turns a flat comment
   thread into recurring themes, concrete action items, and any disagreement
   between reviewers. Callable as a button on the artifact page and as the
   `get_feedback_summary` MCP tool — a real product feature usable from
   either surface, not a demo bolted onto one.
3. **Search re-rank** (`rerankArtifactsForSearch`) — natural-language search
   over the gallery. At this scale (a demo catalog, not a huge corpus) an LLM
   re-rank over title/description/tags is simpler and good enough; embeddings
   infra would be overkill for the timebox.

## Deployment approach

- `apps/web` → Vercel (Next.js zero-config deploy), Postgres via Neon, files
  via Vercel Blob.
- `apps/mcp-server` → Railway, built from the root `Dockerfile` (build context
  is the monorepo root so it can install the shared `core` workspace package;
  see `railway.json`).
- Both read `DATABASE_URL` pointing at the same Neon database.

I verified the whole system end-to-end against a local Postgres container and
a locally built/run copy of the production Docker image before deploying (see
commit history) — publish/list/comment/share all work over the real DB, and
all 5 MCP tools were exercised over the real Streamable HTTP protocol
(`initialize` → `tools/list` → `tools/call`), including the auth-rejection
path. The two things that couldn't be verified locally are the parts that
need real hosted credentials: the Anthropic-backed features (auto-tagging,
feedback summary, search) and file upload to Vercel Blob — both fail with a
clear, expected "missing credential" error locally and are exercised for real
once deployed with actual keys.

## What I'd do next with another week

- Real auth (even simple email-link or GitHub OAuth) instead of a single
  shared workspace, so artifacts and comments have real ownership.
- Artifact version history — right now publishing is one-shot; a "republish"
  that keeps prior versions would make this useful for iterating on a design.
- Move search to embeddings + a vector index once the catalog is large enough
  that an LLM re-rank over the full candidate set stops being cheap.
- Stronger comment moderation/rate-limiting now that comments are
  unauthenticated.
- A proper CSP for the blob-storage origin and a stricter `sandbox` audit if
  HTML artifacts start being embedded anywhere beyond the single detail-page
  iframe.
